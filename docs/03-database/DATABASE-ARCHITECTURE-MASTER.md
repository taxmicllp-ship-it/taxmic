## 1. Database Design Principles

### 1.1 Core Principles

**Multi-Tenant Architecture**
- All tenant-owned data isolated by `firm_id`
- PostgreSQL Row Level Security (RLS) enforces isolation
- Shared database, logical separation
- No cross-tenant data access possible

**Relational Integrity**
- Strict foreign key constraints
- No duplicated relationships
- Cascade deletes where appropriate
- Referential integrity enforced at database level

**Audit Trail**
- All tables include `created_at` timestamp
- All tables include `updated_at` timestamp
- Soft deletes using `deleted_at` for recoverable data
- Activity events track user actions

**Performance**
- Strategic indexes on foreign keys
- Composite indexes for common queries
- Full-text search using PostgreSQL tsvector
- GIN indexes for JSONB and search columns


### 1.2 Standard Fields

**All Tables Must Include:**
- `id` — UUID primary key (generated by database)
- `created_at` — Timestamp of record creation
- `updated_at` — Timestamp of last update

**Soft-Deletable Tables Must Include:**
- `deleted_at` — Timestamp of soft deletion (NULL if active)

**Tenant-Owned Tables Must Include:**
- `firm_id` — Foreign key to firms table (tenant identifier)

### 1.3 Data Types

**Identifiers:**
- Primary keys: UUID
- Foreign keys: UUID
- External IDs (Stripe, etc.): VARCHAR(255)

**Text:**
- Short text (names, emails): VARCHAR(255)
- Medium text (descriptions): TEXT
- Long text (notes, content): TEXT
- Structured data: JSONB

**Numbers:**
- Integers: INTEGER
- Large integers: BIGINT
- Decimals (money): DECIMAL(10, 2)
- Percentages: INTEGER (0-100)

**Dates:**
- Timestamps: TIMESTAMP (with timezone implied)
- Dates only: DATE

**Boolean:**
- Flags: BOOLEAN (default false)

**Network:**
- IP addresses: INET
- Email addresses: VARCHAR(255)

**Enums (PostgreSQL ENUM types):**
- client_status_enum: 'active', 'inactive', 'archived', 'lead'
- client_type_enum: 'individual', 'business', 'nonprofit'
- task_status_enum: 'new', 'in_progress', 'waiting_client', 'review', 'completed'
- task_priority_enum: 'low', 'medium', 'high', 'urgent'
- invoice_status_enum: 'draft', 'sent', 'paid', 'overdue', 'cancelled'
- payment_method_enum: 'stripe', 'check', 'cash', 'wire', 'other'
- payment_status_enum: 'pending', 'completed', 'failed', 'refunded'
- subscription_status_enum: 'trialing', 'active', 'past_due', 'canceled', 'unpaid'
- document_visibility_enum: 'internal', 'client'
- notification_type_enum: 'task_assigned', 'task_completed', 'invoice_sent', 'invoice_paid', 'document_uploaded', 'comment_added', 'user_invited'
- email_event_type_enum: 'sent', 'delivered', 'opened', 'clicked', 'bounced', 'complained'

**ENUM Benefits:**
- Type safety at database level
- Prevents invalid values
- Better performance than VARCHAR with CHECK constraints
- Self-documenting schema

## 2. Naming Conventions

### 2.1 General Rules

**All names use snake_case**
- Tables: `clients`, `client_contacts`, `document_versions`
- Columns: `firm_id`, `created_at`, `updated_at`, `client_id`
- Indexes: `idx_clients_firm_id`, `idx_documents_search`
- Constraints: `fk_documents_client_id`, `uq_users_email`

### 2.2 Table Names

**Singular vs Plural:**
- Use plural for entity tables: `clients`, `users`, `documents`
- Use singular for join tables: `client_contacts`, `user_roles`
- Use descriptive names: `document_versions` not `doc_vers`

**Prefixes:**
- No prefixes for standard tables
- System tables can use descriptive names: `feature_flags`, `webhook_events`

### 2.3 Column Names

**Foreign Keys:**
- Always use pattern: `{table_name}_id`
- Examples: `client_id`, `user_id`, `invoice_id`, `task_id`
- Exception: `firm_id` (tenant identifier)

**Timestamps:**
- Creation: `created_at`
- Update: `updated_at`
- Deletion: `deleted_at`
- Custom: `{action}_at` (e.g., `sent_at`, `paid_at`, `completed_at`)

**Booleans:**
- Use `is_` prefix: `is_active`, `is_current`, `is_system`
- Or descriptive: `enabled`, `verified`, `archived`

**Counts:**
- Use `_count` suffix: `document_count`, `task_count`

**Amounts:**
- Use `_amount` suffix: `total_amount`, `paid_amount`
- Or `_bytes` for storage: `size_bytes`, `total_bytes`

### 2.4 Index Names

**Pattern:** `idx_{table}_{column(s)}`
- Single column: `idx_clients_firm_id`
- Multiple columns: `idx_documents_firm_client`
- Search: `idx_clients_search`
- Unique: `uq_users_email`

### 2.5 Constraint Names

**Foreign Keys:** `fk_{table}_{column}`
- Example: `fk_documents_client_id`

**Unique Constraints:** `uq_{table}_{column(s)}`
- Example: `uq_users_email`, `uq_invoices_firm_number`

**Check Constraints:** `chk_{table}_{condition}`
- Example: `chk_invoices_amount_positive`

## 3. Multi-Tenant Architecture

### 3.1 Tenant Model

**Tenant Definition:**
- Each accounting firm = 1 tenant
- Tenant identifier: `firm_id` (UUID)
- All tenant data isolated by `firm_id`
- PostgreSQL RLS enforces isolation automatically

### 3.2 Tenant-Owned Tables

**All tables containing tenant data MUST include `firm_id`:**

**Auth Domain:**
- `users` — Firm staff members

**CRM Domain:**
- `clients` — Firm's clients
- `contacts` — Client contacts
- `client_contacts` — Join table
- `client_addresses` — Client addresses

**Documents Domain:**
- `folders` — Document folders
- `documents` — Uploaded files
- `document_versions` — File versions
- `document_permissions` — Access control

**Tasks Domain:**
- `tasks` — Task management
- `task_assignments` — Task assignments
- `task_comments` — Task comments

**Billing Domain:**
- `invoices` — Client invoices
- `invoice_items` — Invoice line items
- `payments` — Payment records

**Notifications Domain:**
- `notifications` — User notifications
- `email_events` — Email tracking

**System Domain:**
- `activity_events` — Activity timeline
- `storage_usage` — Storage tracking

### 3.3 System-Level Tables

**Tables WITHOUT `firm_id` (shared across all tenants):**

**SaaS Billing:**
- `plans` — Subscription plans
- `subscriptions` — Firm subscriptions
- `subscription_events` — Billing events

**System:**
- `firms` — Tenant registry
- `roles` — System roles
- `permissions` — System permissions
- `role_permissions` — Role-permission mapping
- `feature_flags` — Feature toggles
- `webhook_events` — Webhook tracking
- `failed_jobs` — Dead letter queue

**Portal:**
- `client_users` — Client portal accounts
- `portal_sessions` — Portal sessions

### 3.4 Row Level Security (RLS)

**Enforcement:**
- RLS enabled on ALL tenant-owned tables
- Database enforces `firm_id` isolation automatically
- Application sets tenant context via session variable
- Impossible to bypass (even with raw SQL)

**Benefits:**
- Security by default
- Cannot forget WHERE clause
- Works with all query types
- Audit compliance (SOC 2, ISO 27001)

## 5. Complete Table Definitions

### 5.1 AUTHENTICATION & AUTHORIZATION

#### TABLE: firms

**Purpose:** Stores accounting firms (tenants)

**Columns:**
- `id` — UUID, primary key
- `name` — VARCHAR(255), firm name
- `slug` — VARCHAR(100), unique URL-safe identifier
- `email` — VARCHAR(255), primary contact email
- `phone` — VARCHAR(50), contact phone
- `address` — TEXT, physical address
- `website` — VARCHAR(255), firm website
- `logo_url` — VARCHAR(500), logo image URL
- `timezone` — VARCHAR(50), default timezone
- `created_at` — TIMESTAMP, creation time
- `updated_at` — TIMESTAMP, last update time
- `deleted_at` — TIMESTAMP, soft delete (NULL if active)

**Foreign Keys:** None

**Indexes:**
- PRIMARY KEY on `id`
- UNIQUE on `slug`
- INDEX on `email`
- INDEX on `deleted_at` (WHERE deleted_at IS NULL)

**Relationships:**
- firms → users (one to many)
- firms → clients (one to many)
- firms → subscriptions (one to one)

**RLS:** Not applicable (system table)

---

#### TABLE: users

**Purpose:** Stores firm staff members

**Columns:**
- `id` — UUID, primary key
- `firm_id` — UUID, foreign key to firms
- `email` — VARCHAR(255), unique per firm
- `password_hash` — VARCHAR(255), bcrypt hash
- `first_name` — VARCHAR(100), user first name
- `last_name` — VARCHAR(100), user last name
- `phone` — VARCHAR(50), contact phone
- `avatar_url` — VARCHAR(500), profile picture URL
- `is_active` — BOOLEAN, account status (default true)
- `email_verified` — BOOLEAN, email verification status (default false)
- `last_login_at` — TIMESTAMP, last login time
- `created_at` — TIMESTAMP, creation time
- `updated_at` — TIMESTAMP, last update time
- `deleted_at` — TIMESTAMP, soft delete (NULL if active)

**Foreign Keys:**
- `firm_id` → firms.id (ON DELETE CASCADE)

**Indexes:**
- PRIMARY KEY on `id`
- INDEX on `firm_id`
- UNIQUE on `(firm_id, email)` WHERE deleted_at IS NULL
- INDEX on `email`
- INDEX on `deleted_at` (WHERE deleted_at IS NULL)

**Relationships:**
- users → firm (many to one)
- users → user_roles (one to many)
- users → tasks (one to many, as assignee)
- users → documents (one to many, as uploader)

**RLS:** Enabled (firm_id isolation)

---

#### TABLE: roles

**Purpose:** Defines system roles for RBAC

**Columns:**
- `id` — UUID, primary key
- `name` — VARCHAR(50), unique role name (owner, admin, staff, contractor, viewer)
- `description` — TEXT, role description
- `is_system` — BOOLEAN, system-defined role (default false)
- `created_at` — TIMESTAMP, creation time

**Foreign Keys:** None

**Indexes:**
- PRIMARY KEY on `id`
- UNIQUE on `name`

**Relationships:**
- roles → role_permissions (one to many)
- roles → user_roles (one to many)

**RLS:** Not applicable (system table)

**System Roles:**
- owner — Full access to everything
- admin — Manage users and settings
- staff — Manage clients, documents, tasks, invoices
- contractor — Limited access to assigned tasks
- viewer — Read-only access

---

#### TABLE: permissions

**Purpose:** Defines granular permissions for RBAC

**Columns:**
- `id` — UUID, primary key
- `name` — VARCHAR(100), unique permission name (e.g., clients.create)
- `resource` — VARCHAR(50), resource type (clients, documents, tasks, etc.)
- `action` — VARCHAR(50), action type (create, read, update, delete, send)
- `description` — TEXT, permission description
- `created_at` — TIMESTAMP, creation time

**Foreign Keys:** None

**Indexes:**
- PRIMARY KEY on `id`
- UNIQUE on `name`
- INDEX on `resource`
- INDEX on `action`

**Relationships:**
- permissions → role_permissions (one to many)

**RLS:** Not applicable (system table)

**Permission Format:** `{resource}.{action}`
- Examples: clients.create, documents.read, invoices.delete, billing.manage

---

#### TABLE: role_permissions

**Purpose:** Maps roles to permissions (many-to-many)

**Columns:**
- `id` — UUID, primary key
- `role_id` — UUID, foreign key to roles
- `permission_id` — UUID, foreign key to permissions
- `created_at` — TIMESTAMP, creation time

**Foreign Keys:**
- `role_id` → roles.id (ON DELETE CASCADE)
- `permission_id` → permissions.id (ON DELETE CASCADE)

**Indexes:**
- PRIMARY KEY on `id`
- UNIQUE on `(role_id, permission_id)`
- INDEX on `role_id`
- INDEX on `permission_id`

**Relationships:**
- role_permissions → role (many to one)
- role_permissions → permission (many to one)

**RLS:** Not applicable (system table)

---

#### TABLE: user_roles

**Purpose:** Assigns roles to users within a firm

**Columns:**
- `id` — UUID, primary key
- `user_id` — UUID, foreign key to users
- `role_id` — UUID, foreign key to roles
- `firm_id` — UUID, foreign key to firms
- `created_at` — TIMESTAMP, creation time

**Foreign Keys:**
- `user_id` → users.id (ON DELETE CASCADE)
- `role_id` → roles.id (ON DELETE CASCADE)
- `firm_id` → firms.id (ON DELETE CASCADE)

**Indexes:**
- PRIMARY KEY on `id`
- UNIQUE on `(user_id, role_id, firm_id)`
- INDEX on `user_id`
- INDEX on `firm_id`
- INDEX on `role_id`

**Relationships:**
- user_roles → user (many to one)
- user_roles → role (many to one)
- user_roles → firm (many to one)

**RLS:** Enabled (firm_id isolation)

**Notes:**
- Users can have multiple roles within a firm
- Same user can have different roles in different firms (if multi-firm support added)

**⚠️ CRITICAL: Invoice Number Generation Strategy**

**Problem:** Concurrent invoice creation can cause number collisions

**Solution: Use PostgreSQL Sequence Per Firm**

```
-- Create sequence table
CREATE TABLE invoice_sequences (
  firm_id UUID PRIMARY KEY REFERENCES firms(id) ON DELETE CASCADE,
  last_number INTEGER NOT NULL DEFAULT 0,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

-- Atomic number generation function
CREATE OR REPLACE FUNCTION get_next_invoice_number(p_firm_id UUID)
RETURNS INTEGER AS $$
DECLARE
  v_next_number INTEGER;
BEGIN
  INSERT INTO invoice_sequences (firm_id, last_number)
  VALUES (p_firm_id, 1)
  ON CONFLICT (firm_id) DO UPDATE
  SET last_number = invoice_sequences.last_number + 1,
      updated_at = NOW()
  RETURNING last_number INTO v_next_number;
  
  RETURN v_next_number;
END;
$$ LANGUAGE plpgsql;

-- Usage in application
SELECT get_next_invoice_number('firm-uuid-here');
```

**Benefits:**
- Atomic operation (no race conditions)
- Per-firm sequences
- Handles concurrent requests
- No gaps in numbering

## 6. Relationship Integrity

### 6.1 One-to-Many Relationships

**firms → users**
- One firm has many users
- Foreign key: users.firm_id → firms.id
- Cascade: DELETE CASCADE (remove users when firm deleted)

**firms → clients**
- One firm has many clients
- Foreign key: clients.firm_id → firms.id
- Cascade: DELETE CASCADE

**clients → documents**
- One client has many documents
- Foreign key: documents.client_id → clients.id
- Cascade: DELETE CASCADE

**clients → tasks**
- One client has many tasks
- Foreign key: tasks.client_id → clients.id
- Cascade: DELETE CASCADE

**clients → invoices**
- One client has many invoices
- Foreign key: invoices.client_id → clients.id
- Cascade: DELETE CASCADE

**invoices → invoice_items**
- One invoice has many line items
- Foreign key: invoice_items.invoice_id → invoices.id
- Cascade: DELETE CASCADE

**invoices → payments**
- One invoice can have many payments (partial payments)
- Foreign key: payments.invoice_id → invoices.id
- Cascade: DELETE CASCADE

**documents → document_versions**
- One document has many versions
- Foreign key: document_versions.document_id → documents.id
- Cascade: DELETE CASCADE

**tasks → task_assignments**
- One task can have many assignments
- Foreign key: task_assignments.task_id → tasks.id
- Cascade: DELETE CASCADE

**tasks → task_comments**
- One task has many comments
- Foreign key: task_comments.task_id → tasks.id
- Cascade: DELETE CASCADE

### 6.2 Many-to-Many Relationships

**clients ↔ contacts**
- Many clients can have many contacts
- Join table: client_contacts
- Foreign keys:
  - client_contacts.client_id → clients.id (CASCADE)
  - client_contacts.contact_id → contacts.id (CASCADE)

**users ↔ roles**
- Many users can have many roles (per firm)
- Join table: user_roles
- Foreign keys:
  - user_roles.user_id → users.id (CASCADE)
  - user_roles.role_id → roles.id (CASCADE)
  - user_roles.firm_id → firms.id (CASCADE)

**roles ↔ permissions**
- Many roles can have many permissions
- Join table: role_permissions
- Foreign keys:
  - role_permissions.role_id → roles.id (CASCADE)
  - role_permissions.permission_id → permissions.id (CASCADE)

**firms ↔ feature_flags**
- Many firms can have many feature flags
- Join table: firm_feature_flags
- Foreign keys:
  - firm_feature_flags.firm_id → firms.id (CASCADE)
  - firm_feature_flags.feature_flag_id → feature_flags.id (CASCADE)

### 6.3 One-to-One Relationships

**firms ↔ subscriptions**
- One firm has one subscription
- Foreign key: subscriptions.firm_id → firms.id (UNIQUE)
- Cascade: DELETE CASCADE

**firms ↔ storage_usage**
- One firm has one storage usage record
- Foreign key: storage_usage.firm_id → firms.id (UNIQUE)
- Cascade: DELETE CASCADE

### 6.4 Self-Referencing Relationships

**folders → folders (parent-child)**

**Mutual Exclusivity:**

## 9. Constraints and Keys

### 9.1 Primary Keys

**All tables:**
- Column: `id`
- Type: UUID
- Generated: gen_random_uuid()
- Constraint: PRIMARY KEY

### 9.2 Foreign Keys

**Naming Convention:** fk_{table}_{column}

**Cascade Rules:**
- DELETE CASCADE: Child records deleted with parent
- DELETE SET NULL: Foreign key set to NULL
- DELETE RESTRICT: Prevent deletion if references exist

**Examples:**
- fk_users_firm_id: CASCADE (users deleted with firm)
- fk_documents_uploaded_by: SET NULL (preserve document if user deleted)
- fk_subscriptions_plan_id: RESTRICT (cannot delete plan if subscriptions exist)

### 9.3 Unique Constraints

**Naming Convention:** uq_{table}_{column(s)}

**Examples:**
- uq_users_firm_email: UNIQUE (firm_id, email) WHERE deleted_at IS NULL
- uq_invoices_firm_number: UNIQUE (firm_id, number) WHERE deleted_at IS NULL
- uq_roles_name: UNIQUE (name)
- uq_webhook_events_event_id: UNIQUE (event_id)

### 9.4 Check Constraints

**Naming Convention:** chk_{table}_{condition}

**Examples:**
- chk_invoices_amounts_positive:
  - total_amount >= 0
  - paid_amount >= 0
  - paid_amount <= total_amount

- chk_feature_flags_rollout_percentage:
  - rollout_percentage >= 0
  - rollout_percentage <= 100

- chk_document_permissions_actor:
  - (user_id IS NOT NULL AND client_user_id IS NULL) OR
  - (user_id IS NULL AND client_user_id IS NOT NULL)

- chk_activity_events_actor:
  - (actor_user_id IS NOT NULL AND actor_client_user_id IS NULL) OR
  - (actor_user_id IS NULL AND actor_client_user_id IS NOT NULL)

### 9.5 Not Null Constraints

**Required Fields:**
- All id columns
- All firm_id columns (tenant-owned tables)
- All created_at columns
- All updated_at columns
- Business-critical fields (name, email, status, etc.)

**Optional Fields:**
- deleted_at (NULL if active)
- Foreign keys to optional relationships
- Descriptive fields (notes, description)

- clients
- contacts
- folders
- documents
- tasks
- task_comments

**Benefits:**

- Audit trail preserved

### 10.2 Hard Delete Tables

**Tables without deleted_at (immutable):**
- document_versions (audit trail)
- activity_events (audit trail)
- webhook_events (audit trail)
- subscription_events (audit trail)
- email_events (audit trail)
- failed_jobs (until resolved)
- audit_logs (if implemented)

**Behavior:**
- Records never deleted
- Immutable audit trail
- Historical data preserved

### 10.3 Cascade Delete Rules

**DELETE CASCADE:**
- Firm deleted → All tenant data deleted
- Client deleted → Documents, tasks, invoices deleted

- Task deleted → Assignments, comments deleted

## 11. Row Level Security (RLS)

### 11.1 RLS Strategy

**Enforcement Level:** Database (PostgreSQL RLS)

**Why RLS:**
- Security by default
- Cannot be bypassed (even with raw SQL)
- Works with all query types (Prisma, raw SQL, admin queries)
- Audit compliance (SOC 2, ISO 27001)

### 11.2 RLS-Enabled Tables

**All tenant-owned tables:**
- users
- clients
- contacts (via client_contacts)
- client_addresses
- folders
- documents
- document_versions
- document_permissions
- tasks
- task_assignments
- task_comments
- invoices
- invoice_items
- payments
- notifications
- email_events
- activity_events
- storage_usage
- user_roles
- firm_feature_flags
- client_users
- portal_sessions

### 11.3 RLS Policies

**Policy Name:** tenant_isolation_policy

**Policy Type:** USING (SELECT)

**Policy Logic:**
```
firm_id = current_setting('app.current_firm_id')::uuid
```

**Application:**
- Middleware sets session variable: app.current_firm_id
- RLS automatically filters all queries by firm_id
- No WHERE clause needed in application code

**Example:**
```
-- Application sets context
SET LOCAL app.current_firm_id = '123e4567-e89b-12d3-a456-426614174000';

-- Query automatically filtered
SELECT * FROM clients;
-- Returns only clients where firm_id = '123e4567-e89b-12d3-a456-426614174000'
```

### 11.4 RLS for Join Tables

**Tables without direct firm_id:**
- contacts
- client_contacts
- document_versions
- task_assignments
- task_comments
- invoice_items

**Policy Logic:**
```
EXISTS (
  SELECT 1 FROM {parent_table}
  WHERE {parent_table}.id = {current_table}.{parent_id}
    AND {parent_table}.firm_id = current_setting('app.current_firm_id')::uuid
)
```

**Example for contacts:**
```
EXISTS (
  SELECT 1 FROM client_contacts cc
  JOIN clients c ON cc.client_id = c.id
  WHERE cc.contact_id = contacts.id
    AND c.firm_id = current_setting('app.current_firm_id')::uuid
)
```

### 11.5 RLS Bypass for System Operations

**Superuser Bypass:**
- Database superuser can bypass RLS
- Used for system maintenance
- Used for cross-tenant analytics

**Application Bypass:**
- Set session variable to NULL for system operations
- Used for background jobs that process multiple tenants
- Requires explicit permission

## 12. Triggers and Functions

### 12.1 Search Vector Triggers

**Purpose:** Automatically update full-text search indexes

**Tables:**
- clients
- contacts
- documents
- tasks

**Trigger:** BEFORE INSERT OR UPDATE

**Function:** update_{table}_search_vector()

**Logic:**
- Concatenate searchable fields
- Apply weights (A = highest, D = lowest)
- Update search_vector column

**Example for clients:**
```
NEW.search_vector := 
  setweight(to_tsvector('english', COALESCE(NEW.name, '')), 'A') ||
  setweight(to_tsvector('english', COALESCE(NEW.email, '')), 'B') ||
  setweight(to_tsvector('english', COALESCE(NEW.phone, '')), 'C');
```

### 12.2 Storage Usage Trigger

**Purpose:** Track storage usage in real-time

**Table:** documents

**Trigger:** AFTER INSERT OR DELETE

**Function:** update_storage_usage()

**Logic:**
- INSERT: Add size_bytes to storage_usage.total_bytes
- DELETE: Subtract size_bytes from storage_usage.total_bytes
- Update document_count
- Update last_calculated_at

**Behavior:**
- Upsert on INSERT (create if not exists)
- Update on DELETE

### 12.3 Updated At Trigger

**Purpose:** Automatically update updated_at timestamp

**Tables:** All tables with updated_at column

**Trigger:** BEFORE UPDATE

**Function:** update_updated_at_column()

**Logic:**
```
NEW.updated_at = NOW();
RETURN NEW;
```

### 12.4 Invoice Status Trigger

**Purpose:** Automatically update invoice status based on payments

**Table:** payments

**Trigger:** AFTER INSERT OR UPDATE

**Function:** update_invoice_status()

**Logic:**
- Calculate total paid_amount for invoice
- If paid_amount >= total_amount: status = 'paid'
- If paid_amount > 0 AND paid_amount < total_amount: status = 'partial'
- Update invoice.paid_at if fully paid

### 12.5 Activity Event Trigger

**Purpose:** Automatically log activity events

**Tables:** clients, documents, tasks, invoices, payments

**Trigger:** AFTER INSERT OR UPDATE OR DELETE

**Function:** log_activity_event()

**Logic:**
- INSERT: Log {entity}_created event
- UPDATE: Log {entity}_updated event
- DELETE: Log {entity}_deleted event
- Extract actor from session variable
